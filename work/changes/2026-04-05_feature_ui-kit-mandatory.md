# Feature: UI Kit obrigatório por projeto

**Date:** 2026-04-05
**Type:** feature
**Status:** spec-approved

---

## Description

Every Scofield-managed project must have a UI Kit as the canonical source of truth for its design system. The UI Kit lives at `specs/ui/UI_KIT.md` and is the single reference for all visual decisions — tokens, typography, colors, and components — used in the project.

The UI Kit is visualized as an HTML artifact generated by the agent from the `UI_KIT.md` file. The artifact has a fixed base template (structure, navigation, categories) that is populated with the items from the file. The artifact never holds its own state — it is always regenerated from the file.

Items in the UI Kit have two statuses: `pending` (proposed, not yet approved) and `approved` (validated by the user, canonical, immutable without explicit user approval).

---

## Expected Behavior

### Status model

- **`pending`** — item has been proposed by the agent but not yet validated by the user. Cannot be used in project implementation.
- **`approved`** — item has been explicitly validated by the user. Can be used in project implementation. Cannot be altered without explicit user approval.

### Hard rules (enforced in CLAUDE.md and .cursorrules)

1. The agent must not implement any UI element in the project unless that element exists in `specs/ui/UI_KIT.md` with `status: approved`.
2. The agent must not alter any item with `status: approved` without explicit user approval in that session.
3. These rules apply to tokens, typography definitions, color values, spacing scales, component specs, and any other item tracked in `UI_KIT.md`.

### Flow per command

#### `/mentor` (setup step)

1. Read existing project context: `specs/ui/_tokens.md`, any existing component or screen specs.
2. Propose base UI Kit items derived from that context: tokens (spacing, border-radius, shadows), typography (font families, sizes, weights), color palette, and fundamental components.
3. Write `specs/ui/UI_KIT.md` with all proposed items at `status: pending`.
4. Generate an HTML artifact rendering all items, with a clear visual distinction for pending items (e.g., amber border, "PENDING" badge).
5. Present the artifact and list all pending items in the prompt output.
6. Ask the user to validate item by item. An affirmative response in chat (e.g., "yes", "approved", "ok") counts as approval for that item.
7. For each approved item: update `status` to `approved` in `UI_KIT.md` immediately (do not batch).
8. Re-render the artifact after each approval so the user sees the current state.
9. At the end of the session, display a summary: N items approved, M items still pending.

#### `/2-plans-and-tasks`

The implementation plan must include an explicit **UI Kit delta** section that lists:
- UI Kit items that need to be created (new components or tokens required by the feature, not yet in `UI_KIT.md`).
- UI Kit items that need to be altered (approved items that the feature requires to change — each must be flagged as requiring user approval before alteration).

This section must appear before any implementation tasks that depend on those items. UI Kit delta tasks are blockers for implementation tasks.

#### `/3a-implement`

For each UI Kit item listed as "create" or "alter" in the plan:

1. Before implementing in project code, render that item in isolation in the HTML artifact. Visually highlight it as pending (same style as mentor step). List the pending item in the prompt output.
2. Stop and wait for explicit user approval.
3. Only after an affirmative response: update `status` to `approved` (or update the item value if it was an alteration) in `UI_KIT.md`, then proceed to implement in project code.
4. If the item was previously `approved` and the plan requires altering it: present the proposed change explicitly (old value → new value), wait for approval, then update.

The agent must not write any project code for a UI element before its corresponding UI Kit item is `approved`.

## Acceptance Criteria

1. `specs/ui/UI_KIT.md` exists and follows the template structure with status fields per item.
2. `CLAUDE.md` contains the UI Kit hard rules: no implementation without `approved` status, no alteration of `approved` items without user approval.
3. `.cursorrules` mirrors the same rules as `CLAUDE.md`.
4. `.claude/commands/mentor.md` includes the UI Kit setup step (steps 1–9 above).
5. `.claude/commands/2-plans-and-tasks.md` requires a UI Kit delta section in every plan.
6. `.claude/commands/3a-implement.md` requires the per-item approval gate before any UI implementation.
7. `.cursor/commands/mentor.md`, `.cursor/commands/2-plans-and-tasks.md`, and `.cursor/commands/3a-implement.md` are identical mirrors of their `.claude/commands/` counterparts.
8. The HTML artifact template is documented or embedded in `UI_KIT.md` or in the mentor command so it can be reliably regenerated.

## Rationale

Scofield's core proposition is that specs are the source of truth. Currently, visual decisions (colors, spacing, components) are tracked loosely across `_tokens.md`, Figma, and implicit agent knowledge. This creates drift: the agent makes visual decisions ad hoc during implementation, and there is no single place to look up what is canonical.

The UI Kit closes this gap for the visual layer. It does for design what `specs/` does for behavior: gives the agent a single canonical source it must consult and must not contradict.

The approval gate at `/3a-implement` is the critical enforcement point. Without it, the hard rules in `CLAUDE.md` would rely on agent self-discipline. With it, the user sees and approves every new visual element before it enters the project — making the UI Kit a living, user-validated artifact rather than a document that drifts from reality.
